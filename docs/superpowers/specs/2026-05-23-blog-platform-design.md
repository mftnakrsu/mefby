# Blog Platform Design

**Status:** Proposed (revised after chatbot removal decision)
**Date:** 2026-05-23
**Author:** Meftun Akarsu (drafted with Claude)

## Context

`mefby-main` is a Vite + React 19 SPA serving a personal portfolio. Content (résumé sections + writing) lives as TypeScript constants in `constants.ts`. One inline essay ("The Silent Variable in Graph RAG") is rendered via a hand-rolled `components/Markdown.tsx` that supports `##`/`###`, paragraphs, lists, and inline `code`/`bold`/`italic` — no images, no code blocks, no math, no embedded components. A `kvcache.cobanov.dev`-style interactive essay (animated SVGs, custom React widgets per figure) cannot be rendered on the current substrate.

The site also currently mounts a floating Gemini-powered chatbot (`components/ChatBot.tsx`) that calls Gemini directly from the client bundle. The owner has decided to **remove the chatbot entirely** — it sits in a gray zone between demo and product. A real AI showcase, when built, will be a standalone project on a separate subdomain (e.g. `chat.meftun.dev`) with proper RAG / multi-turn / tool-use — not a half-feature embedded in the portfolio.

The chatbot decision changes the framework calculus. Without it, the site is pure static content. Earlier draft of this spec picked Next.js because "the chatbot will grow, and Vercel AI SDK is Next.js-shaped." That tiebreaker is gone. For a pure content site, Astro is the better technical fit (zero-JS-by-default, type-safe Content Collections, content-site-native).

The goal is two sequential PRs:

1. **PR1 — chatbot removal + Gemini key cleanup.** Delete all chatbot code from the current Vite codebase. Critically, delete the leaked Gemini API key from Google AI Studio (the key was in the production bundle for an unknown duration; assume it was scraped; removing it from Vercel env vars is not enough — it must be invalidated upstream).
2. **PR2 — Astro migration.** Rebuild the site on Astro 5 to enable rich MDX essays with React islands for interactive figures. The current portfolio design is preserved 1:1.

**Out of scope (deferred to separate projects):**
- Writing any specific essay (Agentic RAG, Odigos retrospective, etc.) — content task, separate effort.
- Any chatbot / AI assistant feature — explicitly a future standalone project on its own subdomain, not part of this codebase.

## PR2 — Astro migration

### Why Astro (revisited after chatbot removal)

Honest comparison for a pure static content site:

| Dimension | Astro | Next.js |
|---|---|---|
| Static résumé page | Zero JS by default; smaller bundle | More JS by default (RSC closes some gap, not all) |
| MDX | Content Collections + Zod schemas — type-safe frontmatter, first-class | `@next/mdx` — works, but boilerplate, no schema |
| Tailwind v4 | `@tailwindcss/vite` plugin, idiomatic | `@tailwindcss/postcss` plugin |
| Interactive essay widget | React island via `client:visible` / `client:idle` — opt-in hydration | Server Components + client components; whole tree is React |
| Build speed | Generally faster | Slower (heavier framework) |
| Vercel deploy | Excellent (Astro is a supported framework with first-class adapter) | Native |
| Future AI features | Possible but second-class | First-class |

**Decision: Astro.** The future AI showcase is going to a separate subdomain (framework decided when that project starts — likely Next.js, but irrelevant here). The framework for *this* repo is purely about content authoring + portfolio rendering. Astro wins on every axis that matters for that.

Migration cost note: Vite → Astro requires more rework than Vite → Next.js because file-based routing, layout system, and component model are all different. Accept the one-time cost in exchange for a substrate that fits the long-term shape of the site.

### Stack

- **Astro 5** (current as of mid-2026)
- **Tailwind CSS 4** via `@tailwindcss/vite`
- **`@astrojs/mdx`** for MDX support
- **`@astrojs/sitemap`** for `sitemap-index.xml`
- **`@astrojs/rss`** for `rss.xml`
- **`@astrojs/react`** for React islands (only used by essays with interactive figures)
- **`remark-math` + `rehype-katex`** + `katex` CSS for math rendering
- **Shiki** for syntax highlighting (built into Astro's MDX support; no extra package)
- **`@fontsource-variable/inter`** for self-hosted Inter (drops the Google Fonts external request)
- TypeScript with `astro check` wired into `npm run build`

### Content model

- **Essays** live as MDX files in `src/content/essays/`. Each has frontmatter validated by a Zod schema in `src/content/config.ts`. Filename → URL: `graph-rag-language.mdx` → `/essays/graph-rag-language`.
- **Résumé data** stays in `constants.ts`, imported by `src/pages/index.astro` and rendered server-side at build time (zero client JS for the résumé page).
- **External writings** (Medium links) stay in `constants.ts` as before — these are pointers, not content.
- **Essay index** at `src/pages/essays/index.astro` uses `getCollection('essays')` to enumerate all MDX files with their typed frontmatter, sorted by date.

### Content Collection schema

```ts
// src/content/config.ts
import { defineCollection, z } from 'astro:content';

const essays = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    date: z.date(),
    tags: z.array(z.string()).default([]),
    draft: z.boolean().default(false),
  }),
});

export const collections = { essays };
```

Adding a new essay = drop in an MDX file with valid frontmatter. The index page auto-picks it up. `draft: true` essays are filtered out in production builds.

### `<Diagram>` component

Anti-goal: a visualization framework. `<Diagram>` is a thin presentation wrapper — frame, figure number, caption styling. The figure itself is any child content (a static `<img>` for animated SVGs, or a React island for interactive widgets).

```astro
---
// src/components/essay/Diagram.astro
interface Props {
  number?: number | string;
  caption?: string;
  source?: string;
  class?: string;
}
const { number, caption, source, class: className = '' } = Astro.props;
const hasFooter = number != null || caption || source;
---

<figure class={`my-10 rounded-xl border border-zinc-200 bg-white ${className}`}>
  <div class="p-6">
    <slot />
  </div>
  {hasFooter && (
    <figcaption class="px-6 py-3 border-t border-zinc-100 bg-zinc-50/40 text-xs text-zinc-600">
      {number != null && <span class="font-mono text-zinc-400 mr-2">Fig {number}</span>}
      {caption}
      {source && (
        <a href={source} class="ml-2 underline text-zinc-400">source</a>
      )}
    </figcaption>
  )}
</figure>
```

**Usage 1 — animated SVG, no JS (default case):**

```mdx
// src/content/essays/agentic-rag.mdx
---
title: "Agentic RAG: Beyond Naive Retrieval"
description: "Routing, decomposition, self-reflection, CRAG, ReAct loops."
date: 2026-06-15
tags: ["RAG", "LLMs", "Agents"]
---
import Diagram from '../../components/essay/Diagram.astro';

<Diagram number="1" caption="Naive (left) vs agentic (right). The agentic side cycles through the critic at least once.">
  <img src="/essays/agentic-rag/01-naive-vs-agentic.svg" alt="" class="w-full" />
</Diagram>
```

SVG lives in `public/essays/agentic-rag/`, contains its own `<style>` block with `@keyframes`. The MDX file renders to static HTML at build time. **Zero runtime JS for this essay.**

**Usage 2 — interactive React widget (rare, opt-in):**

```tsx
// src/components/essay/RetrievalLoop.tsx
import { useState } from 'react';

export function RetrievalLoop() {
  const [step, setStep] = useState(0);
  const steps = ['retrieve', 'generate', 'critique', 'retrieve again', 'answer'];
  return (
    <div>
      <svg viewBox="0 0 800 450" className="w-full">
        {/* node colors / arrow visibility driven by `step` */}
      </svg>
      <div className="flex items-center justify-between mt-3 text-xs text-zinc-500">
        <span className="font-mono">step {step + 1} / {steps.length}: {steps[step]}</span>
        <div className="flex gap-3">
          <button onClick={() => setStep(0)} className="hover:text-emerald-600">reset</button>
          <button
            onClick={() => setStep(s => Math.min(steps.length - 1, s + 1))}
            className="hover:text-emerald-600"
          >
            step →
          </button>
        </div>
      </div>
    </div>
  );
}
```

```mdx
import Diagram from '../../components/essay/Diagram.astro';
import { RetrievalLoop } from '../../components/essay/RetrievalLoop.tsx';

<Diagram number="4" caption="Self-RAG critic loop. Step to advance.">
  <RetrievalLoop client:visible />
</Diagram>
```

`client:visible` is Astro's opt-in hydration directive: React only loads when the component scrolls into view. Other essays without interactive widgets ship **zero React entirely** — they're pure HTML+CSS.

**Selection rule:** animated SVG with CSS keyframes by default (zero JS, fastest, smallest). React island only when the reader needs to drive parameters. Both forms render inside the same `<figure>` frame for visual consistency across essays.

### SEO

- `Layout.astro`: default `<head>` metadata (title template, description, OG defaults). `@fontsource-variable/inter` self-hosts Inter (drops the Google Fonts external request and the FCP penalty).
- Per-essay: frontmatter feeds `<title>`, `<meta name="description">`, `<meta property="og:*">`.
- `astro.config.mjs`: `site: 'https://meftunakarsu.com'`; `@astrojs/sitemap` integration generates `/sitemap-index.xml` at build time.
- `/rss.xml` via `@astrojs/rss` from the essays collection.
- OG images: per-essay generated at build time using `astro-og-canvas` (or `satori`) from a template that takes title + date.
- JSON-LD: `Person` schema in `Layout.astro`, `Article` schema in `EssayLayout.astro`.
